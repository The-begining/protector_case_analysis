## 1. Project Overview

### What This Project Does

This is a **fleet insurance pricing engine** for Protector Insurance. Given a portfolio of 18 clients with ~2,376 vehicles, it:

1. **Ingests** raw data from an Excel workbook (4 sheets)
2. **Classifies** every vehicle into one of 16 underwriting (UW) categories
3. **Prices** fleet insurance using a risk-differentiated burning cost model
4. **Analyses** risk using ML models (Poisson GLM, credibility, bootstrap, anomaly detection)
5. **Presents** everything in an interactive Streamlit dashboard focused on Client 18

### Why Each Part Exists

| Component | Business Question It Answers |
|---|---|
| Pipeline | "What vehicles does each client have, and what type are they?" |
| Pricing | "How much should we charge this client, and why?" |
| ML Models | "How confident are we in the price? What drives risk?" |
| Dashboard | "Show me everything in one place so I can make a decision" |

### Classification Functions (Lines 157–290)

#### The Classification Problem

We have 2,376 vehicles that need UW categories. The challenge:
- ~2,067 vehicles have API data (Egenklass field) → easy to classify
- ~309 vehicles have NO API data → need alternative methods

#### Classification Cascade: Why 4 Methods?

```
Rule-based (1,949 vehicles) → LLM → ML → Heuristic (fallback)
```

Each method is a fallback for the previous one:
1. **Rule-based** is the most accurate (direct mapping from known data)
2. **LLM** can reason about vehicle attributes but requires an API key
3. **ML** learns patterns from rule-based labels but is less accurate
4. **Heuristic** always produces an answer but is the least nuanced

## 4. Pricing Engine — `pricing.py`

### Purpose
Calculate how much to charge a client for fleet insurance, with full transparency into every assumption.

### The Pricing Formula

```
Per UW Category:
    Pure Premium = Vehicles × (Base Frequency × Risk Factor × Age Adj × Freq Adj) × (Avg Severity × Sev Adj)
    Gross Premium = Pure Premium × (1 + Profit Margin + Expense Loading + Reinsurance Loading)

Total Price = Sum of Gross Premium across all categories
```

### Why This Formula?

This is a **burning cost model** — the most common approach in commercial insurance:
- **Burning cost** = use historical claims to predict future costs
- **Risk-differentiated** = don't charge the same rate for a fire truck and a moped
- **With loadings** = the insurer needs to cover expenses and make a profit

The pricing engine gives a **point estimate** (one number). But an underwriter needs to know:
- How confident are we? (bootstrap)
- Should we trust this client's data? (credibility)
- What drives their risk? (Poisson GLM)


**What:** Fits the model, weighting each client by their exposure (vehicle-years).
**Why `sample_weight=exposure`?** A client with 279 vehicles over 5 years has 1,395 vehicle-years of data. Their frequency estimate is much more reliable than a client with 37 vehicles over 2 years (74 vehicle-years). Weighting by exposure tells the model to trust larger clients more.

#### Leave-One-Out Cross-Validation (LOO-CV)

```python
    loo_preds = np.zeros(len(X))
    for tr, te in LeaveOneOut().split(X):
        m = PoissonRegressor(alpha=1.0, max_iter=1000)
        m.fit(X_scaled[tr], y_rate[tr], sample_weight=exposure[tr])
        loo_preds[te] = m.predict(X_scaled[te])
```
**What:** For each of the 18 clients, train a model on the other 17 and predict the held-out one.
**Why LOO-CV?** With only 18 data points, traditional train/test split (e.g., 80/20) would mean training on 14 and testing on 4 — too noisy. LOO-CV uses every possible 17/1 split and averages the results. It's the gold standard for small datasets.
**How to interpret:** If LOO MAE < Baseline MAE, the model learned real patterns. If LOO MAE ≥ Baseline MAE, the features don't help — just using the average would be equally good.

### `credibility_weight(client_freq, portfolio_freq, n_claims, k=50)` — Lines 112–115

```python
def credibility_weight(client_freq, portfolio_freq, n_claims, k=50):
    z = n_claims / (n_claims + k)
    return z * client_freq + (1 - z) * portfolio_freq, z
```
**What:** The Bühlmann credibility formula.
**Why `k=50`?** This is the credibility parameter. It means "you need 50 claims before I trust your data 50%." With k=50:
- 10 claims → Z = 10/60 = 0.17 (17% trust)
- 50 claims → Z = 50/100 = 0.50 (50% trust)
- 191 claims → Z = 191/241 = 0.79 (79% trust)
- 500 claims → Z = 500/550 = 0.91 (91% trust)

**The formula:** `blended = Z × client_freq + (1 - Z) × portfolio_freq`
- If Z = 1.0 → use 100% client data, 0% portfolio
- If Z = 0.0 → use 0% client data, 100% portfolio
- If Z = 0.5 → use 50/50 blend

**Why this matters for pricing:** Raw observed frequency is noisy for small clients. Credibility gives a **stabilized** estimate that's better for long-term pricing.

### `bootstrap_premium(claims, fleet, client, loading, n_boot, seed)` — Lines 142–174

**What:** Simulates 2,000 possible premium outcomes by resampling claims data.

```python
    rng = np.random.RandomState(seed)
```
**Why fixed seed?** Reproducibility. Same seed = same random numbers = same results every time.

```python
    observed_n = len(cc)        # actual number of claims
    amounts = cc["Incurred idx"].values  # actual claim amounts
```
**What:** Extract the real claim count and real claim amounts for this client.

```python
    for _ in range(n_boot):           # 2,000 iterations
        n = rng.poisson(observed_n)   # random claim count
        boot = rng.choice(amounts, size=n, replace=True)  # random severities
        premiums.append(boot.sum() / exposure * loading)   # premium for this scenario
```
**How each iteration works:**
1. **Draw a random claim count from Poisson(observed_n):** If Client 18 had 191 claims, Poisson(191) might generate 175, 203, 191, 182, etc. This simulates "what if luck was different?"
2. **Randomly sample n claim amounts WITH replacement:** From the actual claim amounts, pick n values randomly. Some iterations will pick more expensive claims, some cheaper.
3. **Calculate premium:** Sum of resampled amounts ÷ exposure years × loading factor.

**Why Poisson for claim count?** Poisson distribution models "number of random events in a fixed period" — exactly what insurance claims are.
**Why resample with replacement?** This is the bootstrap method — by resampling, we explore the space of "what could have happened."

```python
    return {
        "mean": float(np.mean(premiums)),
        "ci_5": float(np.percentile(premiums, 5)),
        "ci_95": float(np.percentile(premiums, 95)),
        "premiums": premiums,
    }
```
**How to read the output:**
- **mean** = average premium across all 2,000 simulations
- **ci_5 / ci_95** = 90% confidence interval. "We're 90% sure the true premium is between these two numbers."
- **premiums** = the full array of 2,000 values (used to draw the histogram)

### `detect_claim_anomalies(claims, client)` — Lines 179–196

```python
    q1, q3 = nz["Incurred idx"].quantile(0.25), nz["Incurred idx"].quantile(0.75)
    iqr = q3 - q1
    upper, severe = q3 + 1.5 * iqr, q3 + 3.0 * iqr
```
**What:** IQR (Interquartile Range) method for outlier detection.
**How it works:**
- Q1 = 25th percentile (25% of claims are below this amount)
- Q3 = 75th percentile (75% of claims are below this amount)
- IQR = Q3 − Q1 (the "middle 50%" spread)
- **Outlier threshold** = Q3 + 1.5 × IQR
- **Severe outlier threshold** = Q3 + 3.0 × IQR

**Example:** If Q1=5,000 and Q3=20,000, IQR=15,000.
- Outlier threshold = 20,000 + 1.5×15,000 = 42,500 SEK
- Severe threshold = 20,000 + 3.0×15,000 = 65,000 SEK
- Any claim above 42,500 is flagged; above 65,000 is severely flagged.

**Why IQR?** It's robust to skewed distributions (unlike mean±2σ which assumes normality). Insurance claim amounts are always right-skewed (many small claims, few huge ones), so IQR is the right choice.

```python
    nz["anomaly_score"] = ((nz["Incurred idx"] - q3) / max(iqr, 1)).round(2)
```
**What:** How many IQRs above Q3 this claim is.
**How to read:** Score of 1.5 = exactly at the outlier threshold. Score of 3.0 = at the severe threshold. Score of 5.0 = way beyond anything normal.
